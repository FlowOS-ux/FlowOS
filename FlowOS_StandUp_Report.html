1. Project Overview

What is FlowOS?

FlowOS is a “skip the line” platform — a virtual queue management system. Instead of physically standing in line at a clinic, bank, salon, restaurant, or government office, customers join the queue from their phone and watch their turn approach in real time.

What problem does it solve?

Waiting in physical queues wastes time and is frustrating — you can’t see how long the wait is, how many people are ahead, or when it’s your turn. Businesses also struggle to manage crowds. FlowOS makes waiting transparent (live position + estimated time), remote (join from anywhere), and efficient (businesses run their counters from an app).

Current project goal

Deliver a working, demonstrable end-to-end product: a business can sign up, set itself up, and serve customers; a customer can discover a business, join a queue, and track it live. This is working today in a web browser, talking to a real backend and database. The same code is built to become native Android/iOS apps with no rewrite.

2. Architecture Overview

Think of FlowOS as three layers that talk to each other: the app (what users see), the backend (the brain/rules), and the database (the memory). A real-time channel sits alongside to push live updates.

  • Frontend (the app): One React Native codebase. Runs as a mobile app (Android/iOS) and in the browser (React Native Web). Shows screens, collects input, calls the backend — and asks the backend for the truth rather than holding its own logic.
  • Backend (the brain): Node.js + Express, a clean layered modular monolith. Every request flows: route → validate → check login → check permission → controller → service → repository → database.
  • Database (the memory): MongoDB via Mongoose. Only the repository layer touches it, keeping data access clean.
  • Real-time (the live wire): Socket.IO. On a change, the server sends a tiny “something changed” signal and the app refetches the latest — the “signal → fetch” model.
  • How they communicate: the app calls the backend over REST for all actions/data; the backend reads/writes MongoDB; live updates go out as Socket.IO signals. Push, email, and AI are plug-ins behind interfaces that degrade gracefully if unconfigured.

6. Database Design

Collections created (16): users, businesses, staffMembers, queues, queueEntries, appointments, savedBusinesses, reviews, notifications, deviceTokens, analyticsEvents, supportTickets, aiConversations, categories, refreshTokens, kycRequests.

Relationships (plain terms)

  • A user can own businesses, and be staff at businesses (role: Owner / Manager / Staff).
  • A business has many queues; a queue has many entries (each entry = one customer’s place in line).
  • A business also has reviews, appointments, and analytics events.
  • A user has their own entries, appointments, saved places, reviews, notifications, devices.
  • A user can review or save a business only once (enforced by the database).

How data flows: app request → backend validates & checks permissions → business rules run → repository reads/writes MongoDB → clean JSON answer returns. Any change others care about also fires a real-time signal.

7. Real-Time Flow

FlowOS uses the “signal → fetch” approach: the server sends a small nudge, and the app fetches the fresh details — fast updates, always-correct data.

  1. Customer joins a queue → app opens a Socket.IO connection (verified with their login token).
  2. App subscribes to rooms: their user room, the queue room, and (owners) their business room.
  3. An operator taps “Call next” → backend updates the database (one person WAITING → CALLED).
  4. Backend emits a signal to the queue room and affected user rooms (“queue changed”).
  5. Every customer’s app in that queue refetches position + ETA → screen updates automatically.
  6. On the customer’s turn, the backend also writes a notification and signals their user room → feed + live badge update instantly. (With Firebase on, this also becomes a phone push.)

When position changes: position = people ahead + 1; ETA = people ahead × average service time, where the average self-tunes from real serving times.
